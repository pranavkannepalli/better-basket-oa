Runtime benchmarks and estimates
Date: 2026-07-10
Machine: 12 CPU threads, 31 GiB RAM, RTX 3070 8 GiB
Dataset: 233,214 A rows, 55,516 B rows loaded by current parser
Benchmark setting: --max-workers 8

Summary
-------
Fastest practical path:
  TF-IDF only, no local embeddings, process workers:
    .venv/bin/python run_matcher.py --no-embeddings --worker-mode process --max-workers 8 --checkpoint-every 1000

Default higher-recall path:
  TF-IDF + local embeddings + FAISS CPU + process workers:
    .venv/bin/python run_matcher.py --worker-mode process --max-workers 8 --checkpoint-every 1000

LLM adds time based on number of medium/high rule candidates. In the 1000-row
sample, 87/1000 rows hit medium/high rules, so full run estimate is about
20,290 LLM calls if distribution holds.

LLM estimates
-------------
Hybrid process+thread LLM mode
------------------------------
Process worker mode now supports LLM runs. The CPU-heavy deterministic
retrieval/scoring stage runs in forked process workers. Rows that need LLM
evaluation are handed back to the parent process and finished in a thread pool,
so CPU scoring can continue while API calls are in flight.

TF-IDF + LLM, process workers, 200-row benchmark:
  Command:
    .venv/bin/python run_matcher.py \
      --limit-a 200 \
      --no-embeddings \
      --worker-mode process \
      --max-workers 8 \
      --checkpoint-every 50 \
      --min-confidence 0 \
      --output-dir /tmp/bb-hybrid-process-200

  Observed:
    Total runtime: 16.9s / 200 A rows
    LLM decisions: 19
    Effective CPU cores during CPU-heavy checkpoints: 6.7-7.3
    Final effective cores: 2.9 because the tail waited on LLM calls
    Workers observed: 8

TF-IDF + FAISS embeddings + LLM, process workers, 200-row benchmark:
  Command:
    .venv/bin/python run_matcher.py \
      --limit-a 200 \
      --worker-mode process \
      --max-workers 8 \
      --checkpoint-every 50 \
      --min-confidence 0 \
      --output-dir /tmp/bb-hybrid-emb-process-200

  Setup:
    Reused cached retrieval-index-b.pkl.
    Reused cached B embeddings .npz.

  Observed:
    Total runtime: 23.9s / 200 A rows
    LLM decisions: 19
    Effective CPU cores during CPU-heavy checkpoints: 6.7-7.0
    Final effective cores: 3.3 because the tail waited on LLM calls
    Workers observed: 8

Interpretation:
  LLM runs can now burn CPU during deterministic scoring. Effective cores will
  naturally fall near the end of a run if remaining work is mostly API waiting.

Observed medium/high rule rate in the 1000-row sample:
  87 / 1000 = 8.7%

Estimated full-run LLM calls:
  233,214 * 0.087 = about 20,290 calls

With 8 workers, LLM wall-clock overhead is roughly:
  1s average API call: 20,290 / 8 = 0.70h
  2s average API call: 1.41h
  5s average API call: 3.52h
  10s average API call: 7.04h

    2s API avg: 5.6h
    5s API avg: 7.7h
    10s API avg: 11.2h

Recommendation
--------------
Use TF-IDF only with process workers for fastest full run and high CPU use:
  .venv/bin/python run_matcher.py --no-embeddings --worker-mode process --max-workers 8 --checkpoint-every 1000

Use process workers for LLM runs when you want CPU scoring and API calls overlapped:
  .venv/bin/python run_matcher.py --no-embeddings --worker-mode process --max-workers 8 --checkpoint-every 1000

Use FAISS embeddings with process workers when you want higher recall and high CPU use:
  .venv/bin/python run_matcher.py --worker-mode process --max-workers 8 --checkpoint-every 1000

FAISS details:
  Package: faiss-cpu
  No Docker/container needed.
  Current implementation builds an in-memory FAISS HNSW CPU index each run from
  cached B embeddings. For the current B catalog, that build took 7.0s.
  If faiss-cpu is missing or FAISS_ENABLED=0, the code falls back to NumPy.

Notes
-----
These estimates assume cached B embeddings for embedding paths. A cold embedding
cache must first build B embeddings once, then save them to .npz.

The RTX 3070 is not materially used by the current local embedding path; fastembed
is running through the installed CPU runtime here. The main utilization observed
was CPU.

Process worker mode
-------------------
--worker-mode process uses ProcessPoolExecutor for CPU-bound deterministic
scoring, especially --no-llm runs. The default start method is:
  Linux: fork
  macOS: spawn

You can force it with:
  --process-start-method fork
  --process-start-method spawn

Linux fork shares the already-loaded product/index/FAISS memory copy-on-write
and avoids the Python GIL limiting scoring to about 2 effective cores.

macOS spawn starts fresh Python worker processes. That makes process mode work
on Mac, but startup is slower and memory use is higher because each worker
receives its own product/index state. On the embedding path, each spawned worker
loads the local embedding model, loads cached B embeddings, and builds its own
in-memory FAISS index.

If LLM is enabled, process workers do deterministic scoring and the parent
finishes LLM-needed rows in a thread pool. This avoids forking API clients while
still keeping CPU-heavy work in separate processes.
