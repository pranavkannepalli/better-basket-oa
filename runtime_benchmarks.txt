Runtime benchmarks and estimates
Date: 2026-07-10
Machine: 12 CPU threads, 31 GiB RAM, RTX 3070 8 GiB
Dataset: 233,214 A rows, 55,516 B rows loaded by current parser
Benchmark setting: --max-workers 8

Summary
-------
Fastest practical path:
  TF-IDF only, no local embeddings, process workers:
    .venv/bin/python run_matcher.py --no-embeddings --worker-mode process --max-workers 8 --checkpoint-every 1000

Default higher-recall path:
  TF-IDF + local embeddings + FAISS CPU + process workers:
    .venv/bin/python run_matcher.py --worker-mode process --max-workers 8 --checkpoint-every 1000

LLM adds time based on number of medium/high rule candidates. In the 1000-row
sample, 87/1000 rows hit medium/high rules, so full run estimate is about
20,290 LLM calls if distribution holds.

Recommendation
--------------
Use TF-IDF only with process workers for fastest full run and high CPU use:
  .venv/bin/python run_matcher.py --no-embeddings --worker-mode process --max-workers 8 --checkpoint-every 1000

Use thread workers when LLM is enabled:
  .venv/bin/python run_matcher.py --no-embeddings --max-workers 8 --checkpoint-every 1000

Use FAISS embeddings with process workers when you want higher recall and high CPU use:
  .venv/bin/python run_matcher.py --worker-mode process --max-workers 8 --checkpoint-every 1000

FAISS details:
  Package: faiss-cpu
  No Docker/container needed.
  Current implementation builds an in-memory FAISS HNSW CPU index each run from
  cached B embeddings. For the current B catalog, that build took 7.0s.
  If faiss-cpu is missing or FAISS_ENABLED=0, the code falls back to NumPy.

Notes
-----
These estimates assume cached B embeddings for embedding paths. A cold embedding
cache must first build B embeddings once, then save them to .npz.

The RTX 3070 is not materially used by the current local embedding path; fastembed
is running through the installed CPU runtime here. The main utilization observed
was CPU.

Process worker mode
-------------------
--worker-mode process uses Linux fork + ProcessPoolExecutor. This is intended
for CPU-bound deterministic scoring, especially --no-llm runs. It shares the
already-loaded product/index memory copy-on-write and avoids the Python GIL
limiting scoring to about 2 effective cores.

If LLM is enabled, the pipeline falls back to thread workers because API clients
and disk caches should stay in the parent process/thread pool path.
